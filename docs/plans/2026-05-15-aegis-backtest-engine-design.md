# Aegis Backtest Engine — Design Doc

**Date:** 2026-05-15
**Status:** Draft for approval
**Author:** Ash + Claude (brainstorm)

---

## 1. Purpose

A local Streamlit app that answers two questions empirically across the user's tradeable universe and 5+ years of history:

1. Does **Scoring v1.8 SC_MOMENTUM ≥ 75** actually predict winning trades better than chance?
2. Conditional on SC_MOM ≥ 75, which combinations of the underlying engines (**Flow / Energy / Structure / MP / Elder**) sharpen win rate, expectancy, and R-multiple?

The output is a **combination analyzer** — pick a filter recipe in the UI, get back signal count, win rates, average forward returns, hit-target rate, hit-stop rate, expectancy. Compare two recipes side-by-side.

### Non-goals (v1)

- No portfolio simulation, position sizing, equity curve, or rebalancing.
- No transaction costs or slippage.
- No regime gating (VIX-based no-entry rules).
- No Pipeline Screener replication.
- No SC_POSITION pipeline (BQ + K39 weekly stoch). Scope = SC_MOMENTUM only.
- No earnings-date awareness (FMP earnings calendar pull is deferred — engine defaults to "max" score for earnings proximity).
- No machine-learning auto-discovery of recipes. User picks combinations manually.

## 2. User-facing flow

User double-clicks `run_app.bat` on the desktop. A hidden console launches Streamlit; the browser opens to `http://localhost:8501`.

### Sidebar (left)

- **Universe** dropdown — defaults to the 80-ticker list; can pick "All" or "Custom" (reads `data/universe.txt`).
- **Date range** — start/end date pickers. Defaults: last 5 years.
- **Filter recipe builder:**
  - SC_MOMENTUM ≥ slider (0-100, default 75)
  - Flow ≥ slider (0-100, default 0)
  - Energy ≥ slider (0-100, default 0)
  - Structure ≥ slider (0-100, default 0)
  - MP ≥ slider (0-100, default 0)
  - MP state checkbox group (BUILDING / STRONG / FADING)
  - Elder ≥ slider (0-10, default 0)
  - Cross-up cooldown days (default 21 — re-trigger only after SC_MOM has been below threshold for ≥ this many days)
- **Outcome window** — checkboxes for 5d, 10d, 21d (default all three).
- **Save recipe** button + recipe name field — persists to `recipes.json` for reuse.
- **Run scan** button.

### Main area

- After "Run scan": a metrics card showing **signal count**, **win rate (forward return > 0)**, **win rate (R-multiple ≥ 1)**, **avg fwd return**, **hit-target rate (2:1 R:R)**, **hit-stop rate (2×ATR)**, **expectancy** ($ per 1R risked). One card per outcome window.
- Below the card: a sortable table of all signals (date, ticker, SC_MOM, Flow, Energy, Structure, MP, Elder, fwd_5d, fwd_10d, fwd_21d, hit_target, hit_stop, R_realized).
- Below that: a histogram of forward returns + cumulative-return-of-signals line chart (matplotlib PNG).
- A "Compare to recipe X" dropdown to load a saved recipe and show a side-by-side metrics card.
- "Export CSV" button — dumps the signal table to `output/signals_<timestamp>.csv`.

## 3. Data model

### 3a. Panel parquet — `data/panel_daily.parquet`

Long format, one row per (ticker, date):

| date | ticker | open | high | low | close | volume |

Plus a parallel `data/panel_weekly.parquet` (Friday-aligned) for Pine `request.security("W", ...)` parity.

Plus `data/spy_daily.parquet` (benchmark, separate file for fast joins).

### 3b. Score panel — `data/scores_daily.parquet`

One row per (ticker, date). Generated by running all engines over `panel_daily.parquet`:

| date | ticker | flow_100 | energy_100 | structure_100 | mp_100 | mp_state | elder_score | sc_momentum | sc_m_strong | sc_m_qualified | sc_m_cross_up |

`sc_m_cross_up` = True only on the day SC_MOM crosses above the user's threshold after a cooldown period. This is the primary "is this a signal?" flag.

### 3c. Signal record — produced at scan time

Built by joining the score panel to forward returns + outcome flags from the price panel:

| date | ticker | sc_momentum | flow_100 | energy_100 | structure_100 | mp_100 | mp_state | elder_score | entry_close | atr14_at_entry | stop_price | target_price | fwd_ret_5d | fwd_ret_10d | fwd_ret_21d | hit_target_21d | hit_stop_21d | days_to_outcome | r_realized_21d |

### 3d. Recipe — `recipes.json`

```json
{
  "name": "Strong Momentum + Building MP",
  "sc_mom_min": 75,
  "flow_min": 70,
  "energy_min": 60,
  "structure_min": 0,
  "mp_min": 0,
  "mp_states": ["BUILDING"],
  "elder_min": 6.5,
  "cooldown_days": 21
}
```

### Module responsibilities

- **fmp_client** — single class `FMPClient` with `get_daily_bars(ticker, from_date, to_date)`, `get_weekly_bars(...)`, polite rate limiting (300/min on Starter is generous; we cap at ~250/min). API key from `os.environ["FMP_API_KEY"]`.
- **panel_builder** — orchestrates universe pull. Idempotent: skips tickers already in cache. Writes long-format parquet. Has a "refresh" mode that pulls only new bars since last cached date.
- **engines/*.py** — each file exposes `compute(daily_df: pd.DataFrame, spy_df: pd.DataFrame, weekly_df: Optional[pd.DataFrame]) -> pd.DataFrame` returning the score column(s) keyed by date. Each function has comments mapping every line back to the Pine source line number (per the port spec).
- **score_runner** — loops over universe, calls all engines, joins outputs, writes `scores_daily.parquet`. Heavy compute path. Should support partial refresh.
- **signal_detector** — given scores parquet + threshold + cooldown, returns DataFrame of (date, ticker) signal events.
- **outcome_tracker** — given signal events + price panel, computes forward returns and stop/target outcomes. Pure pandas, vectorised.
- **recipe / metrics / streamlit_app** — UI layer. No business logic beyond filter application.

## 5. Build sequence (priority order)

The brainstorm settled on building scoring engines first. These are the long pole — UI is meaningless without them. Order within engine porting matters: shared utilities first, then leaves, then composite.

1. **Scaffold** (1-2 hours) — repo structure, `.env`, `.gitignore`, `requirements.txt`, `run_app.bat`, this design doc committed.
2. **Data layer** (½ day) — `fmp_client.py` + `panel_builder.py` + universe file. Pull SPY + 5 test tickers (NVDA, XOM, JPM, COST, VRT) for spot-checking. Verify weekly resampling lines up with TV weekly bars (Friday close).
3. **Shared utilities** (½ day) — `utils.py`. Wilder RMA, ATR, RSI, MACD (signal=EMA(9), not SMA), Heikin Ashi (stateful, must loop), linreg, stdev (population, ddof=0). Unit tests against Pine reference values.
4. **Elder engine** (2 hrs) — simplest. Tests the MACD utility.
5. **MP v1.2** (½ day) — exercises Wilder RMA in ADX (critical correctness).
6. **Flow v1.3** (1 day) — multiple sub-components, Heikin Ashi quality, OBV-via-A/D linreg.
7. **Energy v1.3.1** (1 day) — stateful `en_trend_bars` counter (must loop), Bollinger/Keltner squeeze.
8. **Structure v1.5.0** (1.5 days) — the hardest. 3-mode BD detection + DSG-02 latch/decay (stateful loop), weekly trend join, 60-bar consolidation high.
9. **Composite** (1 hr) — Scoring v1.8 weights only, no gating. Trivial once engines exist.
10. **Score runner** (½ day) — loop universe, build `scores_daily.parquet`. Includes cross-up flag computation.
11. **Outcome tracker** (½ day) — vectorised forward-return + stop/target lookup.
12. **Streamlit UI** (1 day) — sidebar + metrics card + signal table + matplotlib chart + compare view.
13. **Smoke test** (½ day) — run end-to-end on 5 tickers, verify recipe filter + metrics produce sensible numbers.

**Total estimate:** 8-10 working days. Each engine is independently testable, so progress is visible early.

## 6. Validation strategy (no TV exports)

Per the project memory: Pine is the spec, FMP is the data, no TV CSV workflow.

For each engine, validation = three checks:

1. **Math review** — Pine source comments map to Python source. Code review compares line-by-line; the comment block in each Python function names the Pine line numbers it implements.
2. **Sanity bounds** — engine output is always in `[0, 100]` (or `[0, 10]` for Elder). No NaN floods. No infinities. Spot-check on NVDA last 60 bars: scores should be high during rallies, low during drawdowns.
3. **Eyeball spot-check (optional, user does this)** — load NVDA in TradingView, note last week's SC_MOM values from the v1.8 dashboard table, compare to Python output for the same dates. Tolerance: ±2 points (loose, because earnings-date data is missing on the Python side, which can shift Structure by up to 10/95 = ~10pts).

If a Python score diverges by more than ±5 points from TV on a stable trend bar (no earnings within 20d), flag it as an engine bug and fix Python.

**No automated TV-CSV-comparison gate.** Build proceeds without it.

## 7. Open risks

- **Earnings proximity gap (Structure scoring)** — Pine reads `earnings.future_time` from TV. FMP earnings calendar exists but pulling it for the universe is extra work. v1 punts: every signal gets the "≥20 days out" max score (10/95). This biases Structure scores upward by up to 10 points on signals that are actually within earnings windows. Note in the report so the user can mentally adjust. Phase 2: pull FMP earnings calendar.
- **Survivorship bias** — universe is current-tradeable names. Stocks that crashed and got delisted aren't in the panel. This biases all win rates upward. Acknowledged. v1 ships with the bias; documenting it loudly in the UI is enough for Phase 1.
- **Look-ahead in weekly resampling** — Pine `request.security("W", ..., lookahead_on)` returns the in-progress weekly bar. Naive Friday resampling in Python can use Friday's bar before it closes. Mitigation: weekly bar at any daily date `t` uses the most recently *completed* week (last bar where day-of-week == Friday and date < t).
- **Heikin Ashi recursion seed** — first HA_open depends on first available bar. Different seed values diverge for ~20 bars then converge. Will use the same seed Pine uses (`(open + close) / 2` of first bar) so the divergence is bounded.
- **Rate limit on Starter plan** — FMP Starter is 300 calls/min. Universe of 80 tickers × (daily + weekly) = 160 calls per full panel build. ~40 seconds. Fine. Refresh mode does it incrementally.
- **Cross-up signal density on choppy stocks** — a stock that oscillates around SC_MOM = 75 could trigger multiple cross-ups per cooldown window. The cooldown rule (re-trigger only after SC_MOM dropped below threshold for ≥ N days) handles this.

## 8. Phase 2 (out of scope for v1, planned)

- FMP earnings-calendar integration → real Structure earnings score.
- VIX integration + regime classification → optional regime filter in recipe builder.
- DSL v1.4 trail tier system → upgrade outcome tracker from fixed 2×ATR stop to dynamic trail.
- SC_POSITION pipeline (BQ engine + K39 weekly stoch).
- Pipeline Screener historical replay.
- Recipe optimization helper (greedy search over engine threshold combinations to maximize expectancy).
- Walk-forward IS/OOS partitioning (currently the analyzer treats the full date range as one set).
- S&P 500 universe expansion.
